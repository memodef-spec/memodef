# `body_ref` OPTIONAL field for sibling body files (v0.2)

**Disposition:** Strategist-pending — filed by memodef-strategist; Director ratification needed before maintainer drafts SCHEMA.md text
**Origin:** Forward-work-items list, trigger fired by [memos/inbox/2026-05-01-1430--orgdef-strategist--memodef-strategist--body-ref-case-2-trigger.openthing](../memos/inbox/2026-05-01-1430--orgdef-strategist--memodef-strategist--body-ref-case-2-trigger.openthing)
**Filed:** 2026-05-01 by memodef-strategist
**Schema impact:** strictly additive; minor version bump 0.1.0 → 0.2.0

## Summary

Add an OPTIONAL `body_ref` field to `memodef:Memo` that references a sibling `.body.md` file co-located with the memo. When `body_ref` is present, `body` SHOULD be a short triage summary (1–3 sentences); the full content lives in the referenced sibling file. Strictly additive — existing inline-body memos remain valid.

## Motivation

Hand-authored operational memos (strategist→implementer handoffs, multi-section briefs, reference material) routinely run 2000+ characters with structured `== Section ==` headers. Composing, editing, and reviewing this content as an inline JSON string in a `memodef:Memo` envelope is genuinely painful: no markdown editor support, escaping noise, no diff readability for content edits. Adopters have surfaced the friction twice:

- **Case 1 (2026-04-27):** brother-canonical-implementor's first memo round-trip — body-as-JSON-string ergonomics flagged as a friction point.
- **Case 2 (2026-05-01):** Thingalog product-strategist proposed a separate `handoffs/` artifact category to escape body-as-JSON-string entirely. Director ratified holding `memos/` as the universal inter-position channel, with the implication that the friction must be resolved at the field-shape level rather than via category proliferation. Cross-spec coordination memo trail: [orgdef-side proposal](https://github.com/orgdef-spec/orgdef/blob/main/proposals/inter-position-communication-convention.md).

The threshold previously logged in the forward-work-items list ("when 2+ humans hand-author memos and the pain becomes operationally relevant") is reached. Operational pain is now driving structural proposals against memodef's design — `body_ref` lets us address the pain inside the existing memo shape rather than fragmenting the memo category.

## Proposed Change

### Schema delta (SCHEMA.md)

Add new SHOULD-section field:

#### `body_ref` (string, relative path)

Path to a sibling file containing the memo's full body content, resolved relative to the memo file's location. When present, `body` SHOULD be a short triage summary (1–3 sentences); the full content lives in the referenced file.

```json
"body_ref": "2026-05-01-1430--alice--bob--quarterly-handoff.body.md"
```

The sibling file SHOULD be a markdown document. The path SHOULD be a bare filename in the same directory as the memo (no `../`, no subdirectories) — this preserves the maildir folder-as-state composability so that `git mv inbox/X.openthing read/X.openthing && git mv inbox/X.body.md read/X.body.md` is a clean atomic pair.

#### `body` field (amendment)

Add to the existing `body` field description: "When `body_ref` is present, `body` SHOULD be a short triage summary (1–3 sentences) for AI-triage purposes — recipients can scan `body` to decide whether to load `body_ref` content. When `body_ref` is absent, `body` contains the full content per the existing convention."

`body` remains MUST (non-empty), regardless of `body_ref` presence.

### Internal consistency rule (SCHEMA.md §Internal consistency rules)

Add: "When `body_ref` is present, the referenced file SHOULD exist at the resolved path. Validators with filesystem access SHOULD verify existence and report a Pass-with-notes finding on broken references. Senders are responsible for ensuring the sibling file is committed alongside the memo."

### File naming convention (RECOMMENDED)

For the sibling file: `<memo-filename-without-.openthing>.body.md`. Co-located in the same directory as the memo. Matches existing `.openthing` filename convention precedent.

### Maildir-lifecycle helper note (README.md)

In the Inbox lifecycle section, add a non-normative note: "When a memo uses `body_ref`, mark-as-read moves the file pair atomically — `git mv inbox/X.openthing read/X.openthing && git mv inbox/X.body.md read/X.body.md`. Implementations MAY provide a small wrapper (`memodef-mark-read X.openthing`) that handles both moves; the spec does not require tooling beyond `git mv`."

## Design rationale (for the two strategist-scope calls)

### Why `body` is required as summary (not optional defer)

The alternative — `body` MAY be absent when `body_ref` is present — was rejected. AI-legibility primacy ([orgdef value, charter line 29-31](../org/memodef-spec.openthing)) ranks AI-to-AI legibility above human convenience and above auditability. The memo envelope is the load-bearing AI-triage artifact: a recipient session scans `from`/`to`/`subject`/`body` to decide whether the memo applies and whether to load further content. Allowing `body` to be empty would force every recipient to load the sibling file just to triage — a per-reader-per-read tax for one-time author convenience. Wrong direction on the value hierarchy.

The reframe: `body` is **always** the AI-triage abstract; `body_ref` is the long-form expansion. Author writes both (small one-time cost); every reader gets cheap triage forever.

### Why co-located sibling (not `bodies/` subdirectory)

The alternative — sibling files in a separate `bodies/` subdirectory — was rejected. It breaks the maildir folder-as-state composability that the architecture-validation arc canonized ([decisions/proposal-2026-04-29-catdef-strategist-architecture-validation.md](../decisions/proposal-2026-04-29-catdef-strategist-architecture-validation.md)). Co-location keeps the file pair atomic in the receiver's mental model, on disk, and in maildir lifecycle moves. The "directory pollution" objection is moot: `inbox/` directories are designed to be browsed; one extra `.body.md` next to a `.openthing` is not pollution. The atomic-move wart is mechanically trivial — a 3-line shell helper or a 2-line tooling wrapper. One operational wart is cheaper than splitting the file pair across directory trees.

## Backward Compatibility

Strictly additive. No existing memodef:Memo artifact becomes invalid. All v0.1 memos remain conformant under v0.2. Implementations that do not implement `body_ref` continue to work — they read `body` as the full content as before; `body_ref` is silently ignored. Implementations that do implement `body_ref` SHOULD prefer `body_ref` content when both are present, with `body` available as triage summary.

The legacy `x.memo.*` form ([SCHEMA.md §Backward-compatible legacy form](../SCHEMA.md#backward-compatible-legacy-form-xmemo-extensions-on-a-generic-catdef-thing)) gets a parallel `x.memo.body_ref` extension by the same shape, for the same reasons. (Inline note: the typed-vs-legacy form question is parked per [decisions/proposal-2026-04-29-catdef-strategist-architecture-validation.md](../decisions/proposal-2026-04-29-catdef-strategist-architecture-validation.md) and is not in scope for this proposal.)

## Conformance Tests

To add to `conformance/`:

- **`valid_memos/with_body_ref_summary.openthing`** — canonical pair: memo with `body` as 2-sentence summary + `body_ref` pointing to sibling file. Sibling file `with_body_ref_summary.body.md` committed alongside.
- **`valid_memos/body_only_no_body_ref.openthing`** — confirms v0.1 inline-body shape remains conformant under v0.2 (regression guard).
- **`invalid_memos/body_ref_with_empty_body.openthing`** — `body_ref` present but `body` absent or empty — fails the SHOULD-have-summary rule (Pass-with-notes per the SHOULD level, but flagged distinctly).
- **`invalid_memos/body_ref_subdirectory.openthing`** — `body_ref` value contains a path separator (`bodies/X.body.md` or `../foo.body.md`) — fails the bare-filename SHOULD rule.

The dangling-reference case (`body_ref` pointing to a file that does not exist) is filesystem-state-dependent and harder to fixture; calling out as a validator implementation choice rather than a static fixture.

## Alternatives Considered

- **Sibling subdirectory (`bodies/X.body.md`) instead of co-located.** Rejected — breaks maildir composability. Detailed in Design rationale.
- **Body-MAY-be-absent when body_ref is present.** Rejected — violates AI-legibility primacy. Detailed in Design rationale.
- **Embed body content as a quoted multiline string with explicit newline literals.** Pre-existing situation; the friction is exactly what motivates this proposal.
- **External URL reference (`body_url`) instead of sibling file.** Rejected — requires network, breaks the "no required infrastructure beyond filesystem + git" red line ([charter line 53-56](../org/memodef-spec.openthing)). Adopters MAY use `x.<domain>.body_url` extensions for this case privately; not a core-spec concern.
- **Defer to memodef v0.3 or later.** Rejected by trigger logic — operational pain is real and bounded; deferring forces adopters to either bear the JSON-string cost or invent local conventions that fragment.
- **Different field name (`body_path`, `body_file`, `external_body`).** `body_ref` matches semver-style coordinate-noun convention used in roledef/orgdef (`derived_from.url`, `recommended_roles[].role`) — neutral on transport (file or future URL), unambiguous, short. Open to bikeshed.
- **Bundle this with form-migration (typed vs legacy).** Rejected — orthogonal concern, adds scope without benefit.

## Open Questions

For Director ratification:

1. Field name: `body_ref` accepted, or prefer `body_path` / `body_file` / other?
2. Co-location SHOULD vs MUST: this proposal lands SHOULD (consistent with the Inbox lifecycle convention's RECOMMENDED level). Promote to MUST if v0.2 should be stricter?
3. Validator existence-check: SHOULD (Pass-with-notes on broken reference) per this proposal — or MUST (Fail on broken reference)?
4. Atomic-move helper: leave to adopters (this proposal's posture), or commit a reference shell helper alongside the spec?
5. Forward direction: should v0.3 consider extending to `attachments_ref` (multiple sibling files)? Out of scope for v0.2 but worth flagging if the pattern looks generalizable.
